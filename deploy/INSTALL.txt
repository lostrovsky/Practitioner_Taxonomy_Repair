============================================================
Practitioner Taxonomy Repair -- Install / Run
============================================================

One-off remediation tool. For each NPI it considers, re-fetches NPPES
live and either:
  * stages an amend (NPPES primary differs from master's primary, or
    NPPES knows a code master doesn't), OR
  * records status='skipped' (master already matches NPPES).

Runs out-of-band from the daily Claim Provider Data Pipeline. Installs
as a sibling folder next to your existing Claim_Provider_Data_Extractor
and Claim_Provider_Data_Loader.

============================================================
PREREQUISITES
============================================================
* Java 21 JRE or JDK on PATH (or via JAVA_HOME).
* SQL Server reachable, with cpe_master populated by the daily pipeline
  AND read access to [HRDW_REPLICA].[PAYOR_DW].[PROVIDER_TAXONOMY] (the
  same taxonomy-name lookup source the pipeline's sp_resolve_taxonomy_names
  uses; overridable via install.config TAXONOMY_LOOKUP_TABLE for
  non-default environments).
* An existing Claim Provider Data Pipeline install: specifically the
  Claim_Provider_Data_Loader\generic-hrp-ws-call.jar must already be
  deployed at <base>\Claim_Provider_Data_Loader\.
* sqlcmd installed (path supplied via install.config SQLCMD_PATH).
* PowerShell 5.1+ (Windows).

============================================================
STEP 1: EXTRACT
============================================================
Extract the release zip to any temporary directory -- NOT on top of
your existing pipeline install. The installer will create the actual
install folder. Example:

    C:\temp\ptr_v1.6.0\

Resulting layout:

    install.ps1                  (installer)
    install.config               (you edit this in STEP 2)
    INSTALL.txt                  (this file)
    version.txt
    run_repair.ps1               (orchestrator; installer copies into repair folder)
    practitioner-taxonomy-repair-<ver>-jar-with-dependencies.jar
    sql\
        create_cpe_repair_objects.sql
    calls\
        practitioner_taxonomy_repair\
            practitioner_taxonomy_repair.properties
            practitioner_taxonomy_repair.report.json
            practitioner_taxonomy_repair.sql.json

============================================================
STEP 2: EDIT install.config
============================================================
Open install.config in the extracted temp directory and fill in:

  DB_URL                      JDBC connection string for SQL Server
  DB_USER, DB_PASSWORD        DB credentials (must have SELECT on cpe_master.*,
                              SELECT on the taxonomy lookup table -- defaults to
                              [HRDW_REPLICA].[PAYOR_DW].[PROVIDER_TAXONOMY] --
                              and INSERT/UPDATE on cpe_repair.*)
  WS_BASE_URL                 HRP connector base URL (no trailing slash)
  CONNECTOR_ADMIN_PASSWORD    Password for the connector_admin WS user
  LOG_ONLY                    true (recommended for first run) or false
  WS_RETRY_*                  retry settings (defaults are fine)
  SQLCMD_PATH                 full path to sqlcmd.exe

Optional:
  NPI_QUERY                   custom SELECT for the auto-derive path
                              (returns one column of NPIs; verbatim)
  DB_MASTER_SCHEMA / DB_REPAIR_SCHEMA
                              schema overrides (defaults: cpe_master / cpe_repair)
  TAXONOMY_LOOKUP_TABLE / TAXONOMY_CODE_COLUMN / TAXONOMY_NAME_COLUMN
                              override the taxonomy-name lookup. Defaults match
                              the daily pipeline: [HRDW_REPLICA].[PAYOR_DW].[PROVIDER_TAXONOMY]
                              with PROVIDER_TAXONOMY_CODE / PROVIDER_TAXONOMY_NAME

Tip: most values can be copy-pasted from your daily pipeline's
env.properties.

============================================================
STEP 3: RUN install.ps1
============================================================
From the extracted temp directory:

    cd C:\temp\ptr_v1.6.0
    .\install.ps1

You'll be prompted for:

  Installation directory      The BASE folder that already contains
                              Claim_Provider_Data_Extractor\ and
                              Claim_Provider_Data_Loader\. The installer
                              will create Practitioner_Taxonomy_Repair\
                              as a sibling.
  (if folder exists)          Confirm overwrite (y/N).
  Apply DDL now?              y to run sql\create_cpe_repair_objects.sql
                              against your DB now; N to do it manually later.

The installer:
  * Creates <base>\Practitioner_Taxonomy_Repair\
  * Copies the jar, run_repair.ps1, install.ps1, install.config, version.txt,
    sql\create_cpe_repair_objects.sql there
  * GENERATES <base>\Practitioner_Taxonomy_Repair\env.properties from
    install.config (used by the loader at run time)
  * GENERATES <base>\Practitioner_Taxonomy_Repair\PractitionerTaxonomyRepair.properties
    from install.config (used by the repair jar directly)
  * Substitutes SQLCMD_PATH into the deployed run_repair.ps1
  * Copies the call folder to
    <base>\Claim_Provider_Data_Loader\practitioner_taxonomy_repair\
    (backing up any existing one to a timestamped sibling first)
  * Optionally applies the DDL via sqlcmd

============================================================
STEP 4: RUN THE REPAIR
============================================================
Everything from here on is operator-facing -- run from the installed
repair folder, not the extracted temp.

    cd <base>\Practitioner_Taxonomy_Repair

Typical first run (LOG_ONLY mode -- inspect SOAP envelopes; no HRP calls):

    .\run_repair.ps1 -NpiFile pilot.txt -DryRun
        -> jar stages nothing, logs what it would stage. No loader call.

    .\run_repair.ps1 -NpiFile pilot.txt
        -> jar stages a run. Loader runs (LOG_ONLY honors env.properties).
           Captures RUN_ID, then invokes the loader. Watch the SOAP in the
           log; verify it looks right.

When ready for real:

    # Set LOG_ONLY=false in env.properties (or pass -LogOnlyOverride for one run)
    .\run_repair.ps1 -NpiFile pilot.txt
        -> Real amends pushed to HRP. Spot-check the affected practitioners
           in HRP.

For the full population (no --npi-file -> auto-derive query, or NPI_QUERY
override if set in install.config):

    .\run_repair.ps1 -Description "Production repair run"

Resume mode (re-run only the loader against an existing run_id; useful
after a mid-run loader failure):

    .\run_repair.ps1 -RunId 7

============================================================
STEP 5: VERIFY
============================================================
After a real (non-LOG_ONLY) run:

    SELECT status, COUNT(*)
    FROM cpe_repair.practitioner_repair
    WHERE run_id = <n>
    GROUP BY status;

Expected: 'loaded' for everything the loader pushed; 'skipped' for
practitioners that already matched NPPES (no amend was needed); 'failed'
for any HRP rejections (investigate via error_message).

Spot-check 5-10 random repaired practitioners in HRP -- confirm primary
taxonomy now matches NPPES live.

============================================================
TROUBLESHOOTING
============================================================
* "install.config not found" -- you ran install.ps1 from outside the
  extracted directory. cd into the extracted folder first.

* "Loader install not found at: <base>\Claim_Provider_Data_Loader" --
  the path you supplied as installation directory doesn't contain the
  existing pipeline siblings. This installer is an add-on; install the
  daily pipeline first, or point at the correct base directory.

* "Repair jar not found" or "Loader jar not found" from run_repair.ps1
  -- run install.ps1 first. run_repair.ps1 expects to live in the
  installed Practitioner_Taxonomy_Repair folder next to the jar, with
  ..\Claim_Provider_Data_Loader\generic-hrp-ws-call.jar reachable.

* "Database connection failed (PASSWORD APPEARS TO BE EXPIRED ...)" --
  rotate the DB password on SQL Server, then update install.config and
  re-run install.ps1 to regenerate env.properties + the repair jar's
  properties file with the new password.

* Loader returns 401/403 -- the connector_admin WS password is wrong,
  or HRP is rejecting the <maintenanceReasonCode>. The current template
  uses 'PractitionerCreateReason / 1' which is a known TODO for a
  taxonomy-overlay amend; see TODO.md.

* NPPES timeouts -- sequential lookups at ~500ms each. A 1000-NPI batch
  takes 8-10 minutes; nothing's wrong if the jar is slow.

============================================================
ROLLBACK / UNINSTALL
============================================================
1. Delete <base>\Practitioner_Taxonomy_Repair\
2. Delete <base>\Claim_Provider_Data_Loader\practitioner_taxonomy_repair\
   (and the timestamped .bak sibling if you don't need it)
3. (Optional) Drop the cpe_repair schema:
       DROP SCHEMA cpe_repair;
   (Will fail if the schema contains objects -- drop those first or
   leave the schema in place; it's harmless.)

cpe_master, cpe_load, cpe and the daily pipeline are untouched
throughout. This tool reads cpe_master.* (read-only) plus the
configured taxonomy lookup table (defaults to the pipeline's
[HRDW_REPLICA].[PAYOR_DW].[PROVIDER_TAXONOMY], read-only) and
writes only to cpe_repair.
